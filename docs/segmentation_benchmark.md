# ON-Harmony Segmentation Benchmark

**Purpose:** Define the complete design of the downstream segmentation benchmark used to evaluate MRI synthesis augmentation methods. This document is the authoritative reference for anyone implementing, reproducing, or extending the benchmark.

---

## 1. Goal and Scope

Train a brain segmentation model on T1w data only, with different synthesis augmentation methods applied on-the-fly. Evaluate generalization to unseen subjects, unseen scanners, and all available MRI contrasts. The methods under comparison are:

| Identifier | Augmentation |
|---|---|
| `baseline` | None — raw T1w, standard nnUNet intensity augmentation |
| `v26_6` | V26_6SignedAlphaTargetGenerator (K-means + Voronoi + signed alpha remap) |
| `synthseg_a` | BrainGenerator mode A (dense label sampling) |
| `synthseg_b` | BrainGenerator mode B (EM mixture) |

All methods use a single nnUNet dataset and shared preprocessing. Results live in separate timestamped directories. The framework is nnUNet v2 exclusively — no MONAI/Lightning.

---

## 2. Dataset: ON-Harmony

- **20 subjects**, each scanned at multiple sites and scanner types (travelling-heads design)
- **165 T1w sessions** total (some subjects have repeat visits to one scanner)
- **8 scanner families:** NOT1ACH, NOT2ING, NOT3GEM, NOT4GEP, OXF1PRI, OXF2PRI, OXF3TRI, OXF4GEP
- **Contrasts per session:** T1w, T2w/FLAIR (anat), bold (func), DWI AP+PA (dwi), EPI AP+PA (fmap), GRE echo-1/echo-2 mag+phase (swi)
- **SynthSeg GT labels:** `derivatives/synthseg_masks/` — full FreeSurfer label maps from SynthSeg v1.0 run on all 165 T1w sessions. Contains 32 FreeSurfer label IDs; remapped to 7 consecutive classes (see §5).

---

## 3. Train / Validation / Test Split

### 3.1 Test Scanner Exclusion

Two scanner families are **permanently reserved for test**. They must not appear anywhere in training or validation, regardless of subject:

- **Test scanners:** `NOT1ACH`, `OXF1PRI` (all visit codes, e.g. NOT1ACH001 through NOT1ACH006)
- **Train/val scanners:** `NOT2ING`, `NOT3GEM`, `NOT4GEP`, `OXF2PRI`, `OXF3TRI`, `OXF4GEP`

Sessions of train/val subjects at NOT1ACH or OXF1PRI are **discarded** — never used for anything.

### 3.2 Test Subject Exclusion

20% of subjects (4 out of 20) are held out for test. Subject selection is deterministic (seeded shuffle by subject ID, seed=12345). The same 4 subjects are used regardless of method or fold.

**Test set =** 4 held-out subjects × {NOT1ACH, OXF1PRI} sessions × all available contrasts (T1w, T2w/FLAIR, bold, DWI, EPI, GRE).

This double isolation (scanner + subject) eliminates all data leakage paths.

### 3.3 4-Fold Cross-Validation

- 16 subjects remain for train/val
- **Subject-level isolation only** (not scanner-level — that would lose too much data)
- 4 folds: fold k uses 4 subjects as val, 12 as train
- Each subject's T1w sessions across all 6 train/val scanners are assigned together (a subject is never split across train and val within a fold)
- Fold assignment: deterministic stratified split by subject ID

**Result:** ~12 subjects × 6 scanners × T1w ≈ 72 training cases per fold; ~4 subjects × 6 scanners × T1w ≈ 24 val cases per fold.

The split is defined once in `data/splits/onharmony_splits.json` by `scripts/nnunet_onharmony/00_create_splits.py`. This file is the single source of truth for all experiments.

## 6. GT Labels for Non-T1w Test Contrasts

SynthSeg was run on T1w only. For evaluating segmentation on T2w, FLAIR, bold, DWI, GRE, EPI, we need GT labels in each contrast's native space.

**Method: ANTs SyN registration** — register each contrast to the subject's T1w (or vice-versa), then apply the transform to the SynthSeg label map using nearest-neighbour interpolation.

```bash
# Register contrast → T1w space
antsRegistrationSyN.sh -d 3 -f {T1w.nii.gz} -m {contrast.nii.gz} \
    -o {out_prefix} -t s -n 8

# Apply transform to label map (nearest-neighbour, no interpolation of labels)
antsApplyTransforms -d 3 -i {T1w_synthseg_7class.nii.gz} \
    -r {contrast.nii.gz} -o {registered_seg.nii.gz} \
    -n NearestNeighbor \
    -t {out_prefix}1Warp.nii.gz -t {out_prefix}0GenericAffine.mat
```

### 6.1 Registration QC

Likely-failed registrations are flagged automatically. Flag criterion: **Normalized Mutual Information (NMI)** between the registered T1w and the target contrast image. Computed with SimpleITK. Expected range ≈ 0.30–0.80 for well-aligned brain MRI; registrations below **0.30 NMI** are flagged.

All cases are logged to `data/splits/test_registration_qc.csv` (columns: subject, scanner, contrast, NMI, flagged). Flagged cases are reviewed manually before evaluation. Additionally: inspect for degenerate segmentations (e.g., CSF > 30% of brain voxels, any class = 0 voxels) as a second QC check.

### 6.2 4D Contrast Preprocessing

Before registration and inference, extract a 3D volume from each 4D modality:

| Modality | Extraction |
|---|---|
| bold | Temporal mean across all frames |
| DWI | b=0 volume from AP direction (first b=0 shell) |
| EPI (fmap) | First volume of AP direction |
| GRE | Echo-1 magnitude only |

## 7. Custom Data Loader: Full-Volume On-the-Fly GPU Synthesis

### 7.1 Motivation

nnUNet's default pipeline crops patches **inside the data loader**, before the transform pipeline. Transforms therefore receive patches, not full volumes. Our synthesis methods (V26_6 in particular) require the full brain volume for correct histogram analysis, K-means parcellation, and Voronoi sub-region computation. This cannot be approximated at patch level.

Solution: **override `generate_train_batch()` in `nnUNetDataLoaderBase`** to return full preprocessed volumes (no crop). Synthesis and cropping then happen inside `train_step()` on GPU, after the data arrives in the training thread.

### 7.2 Data Flow

`num_workers=0` is required: synthesis runs on GPU in the main thread; worker processes don't share the CUDA context.

```
Prefetch thread (CPU only — shares Python process, no CUDA calls):
  case_key = sample_next_case_from_fold()
  data_np, seg_np = ram_cache[case_key]    # microseconds (RAM, no disk)
  data_pt = torch.from_numpy(data_np).pin_memory()   # fast H2D later
  seg_pt  = torch.from_numpy(seg_np).pin_memory()
  prefetch_queue.put({'data': data_pt, 'target': seg_pt, 'keys': [case_key]})

Main training thread (generate_train_batch, num_workers=0):
  return prefetch_queue.get()              # full volume, already pinned

Main training thread (train_step, GPU):
  batch  = next(dataloader)               # from generate_train_batch above
  data   = batch['data'].to(device, non_blocking=True)    # (1, C, D, H, W)
  target = batch['target'].to(device, non_blocking=True)  # (1, 1, D, H, W)
  t0     = perf_counter()
  synth  = synthesize_on_gpu(data, target)   # (1, C, D, H, W) re-normalized
  aug_latency_ms += (perf_counter() - t0) * 1000
  patches, patches_seg = random_crop(synth, target, initial_patch_size, n=2)
  patches_cpu = patches.cpu()
  patches_seg_cpu = patches_seg.cpu()
  transformed = train_transforms(image=patches_cpu, segmentation=patches_seg_cpu)
  data_aug   = transformed['image'].to(device, non_blocking=True)
  target_aug = [t.to(device, non_blocking=True) for t in transformed['segmentation']]
  output = network(data_aug)
  loss = criterion(output, target_aug)
  loss.backward() ...
```

### 7.3 Normalization (Critical)

nnUNet's `ZScoreNormalization` normalizes the image. For Dataset030, the nnUNet planner auto-selected **`use_mask_for_norm = False`** — normalization is computed over **all voxels** in the cropped volume (including skull). The T1w images in ON-Harmony are not skull-stripped, so skull voxels carry non-zero intensity and are included in the z-score statistics. Background outside the brain bounding box (if any) is set to exactly 0 by `image[seg < 0] = 0`, but since the SynthSeg mask covers the entire brain extent (all seg values ≥ 0), this line has no effect.

The preprocessed `.npz` files therefore have: all voxels within the brain bounding box in z-score space (skull included), ≈ N(0,1) globally.

**Inside `synthesize_on_gpu()`:**

```
Step 1 — identify brain mask using GT segmentation (tissue classes 1–6):
    brain_mask = (seg[0, 0] > 0)             # background class 0 = skull/air; exclude it
    # Note: seg is the 7-class GT label loaded alongside data (nnUNet preprocessed)

Step 2 — denormalize to [0, 1] for generator:
    brain_vals = image[0, 0][brain_mask]
    vmin, vmax = brain_vals.min(), brain_vals.max()
    image_01 = zeros_like(image)
    image_01[0, 0][brain_mask] = (brain_vals - vmin) / (vmax - vmin + ε)
    image_01[0, 0][~brain_mask] = 0.0        # background stays 0

Step 3 — apply synthesis:
    synth_01 = generator(image_01, hist_module, labels=seg)   # output in [0, 1]

Step 4 — re-normalize output to z-score within brain:
    synth_brain = synth_01[0, 0][brain_mask]
    synth_z = zeros_like(synth_01)
    synth_z[0, 0][brain_mask] = (synth_brain - synth_brain.mean()) / (synth_brain.std() + ε)
    synth_z[0, 0][~brain_mask] = 0.0        # background stays exactly 0

Output: synth_z — same statistical distribution as the preprocessed T1w input.
         Background = 0, brain ≈ N(0,1). Ready for nnUNet's forward pass.
```

nnUNet's `MaskImageTransform` (last transform in the pipeline) re-zeroes background as a safety net and is compatible with this convention.

### 7.4 Disabled nnUNet Intensity Augmentations

When `synth_prob=1.0`, nnUNet's built-in intensity augmentations are **disabled** because they would compound with the synthesis, adding random distortions on top of intentional contrast variation. The trainer's `get_training_transforms()` override removes:
- `GaussianNoiseTransform`
- `GaussianBlurTransform`
- `MultiplicativeBrightnessTransform`
- `ContrastTransform`
- `GammaTransform` (both instances)

**Kept:**
- `SpatialTransform` (rotation, scaling, elastic deformation)
- `MirrorTransform`
- `SimulateLowResolutionTransform` — **explicitly kept** because it helps the model generalize to low-resolution test modalities (bold at 3mm upsampled to 1mm appears blurry; this augmentation bridges that gap during training)
- `MaskImageTransform` (background zeroing)

For `baseline`: all nnUNet default augmentations are kept unchanged.

### 7.5 Batch Size and Iteration Count

The custom data loader returns one full volume per load operation (batch_size=1 at the loader level). `train_step()` crops 2 patches from each synthesized volume, producing an effective model batch size of 2. The number of gradient update steps per epoch remains nnUNet's default (250 steps × 500 epochs = 125k total updates). Log `aug_latency_ms` per batch to WandB so the synthesis overhead is visible.

## 8. nnUNet Split File — Anti-Contamination Protocol

nnUNet reads `splits_final.json` from `nnUNet_preprocessed/Dataset030/` at training start. If the file is absent, nnUNet silently generates a 5-fold random split. **CRITICAL:** if `fold >= len(splits)`, nnUNet also silently generates a random 80:20 split rather than raising an error.

Our custom base trainer overrides `do_split()` to:
1. Load `splits_final.json` — raise `FileNotFoundError` (not generate) if absent
2. Assert `len(splits) == 4` — raise `ValueError` if count is wrong
3. Assert `self.fold < 4` — raise `ValueError` rather than falling back to random split
4. Assert `set(tr_keys) ∩ set(val_keys) == ∅` — raise `ValueError` on overlap
5. Assert all tr_keys and val_keys belong to the expected 16 train/val subjects — raise on test subject contamination
6. Log the exact subject list to `{results_dir}/fold_{fold}/split_verification.json` at training start

This guarantees that no experiment can accidentally use a wrong split, and that every trained model has a verifiable audit trail of its exact train/val subjects.

- `train/lr`

**Images:**
- First 10 epochs: log every epoch (for normalization and augmentation debugging)
- Thereafter: every 10 epochs

Each image log contains a 4-panel mid-axial slice grid:
1. Raw T1w (before synthesis) — z-score image displayed with per-slice 2nd–98th percentile normalization within brain pixels
2. Synthesized T1w (after synthesis, before spatial aug) — same display normalization
3. GT segmentation overlay (7-class colormap)
4. Model prediction overlay (same colormap)

**Display normalization rule:** extract 2D slice → compute `p2, p98 = quantile(nonzero_pixels, [0.02, 0.98])` → rescale to [0, 1]. Never use global clipping or fixed z-score clip bounds — these hide contrast variation that is the purpose of the augmentation.

## 13. Test Set Evaluation

### 13.1 Inference

Run `nnUNetv2_predict` on test images from the raw BIDS directory. For each method, ensemble predictions from all 4 folds:

```bash
# Per fold: generate predictions
nnUNetv2_predict -i {test_images_dir} -o {pred_fold_k_dir} \
    -d Dataset030 -c 3d_fullres -f {k} -tr nnUNetTrainerOnHarmony{Method}

# Ensemble across folds
nnUNetv2_ensemble -i {pred_fold_0} {pred_fold_1} {pred_fold_2} {pred_fold_3} \
    -o {pred_ensemble_dir} -np 8
```

nnUNet automatically resamples test images to the planned spacing (1mm from T1w training data).

### 13.2 Resampling Behavior Across Contrasts

All test images are resampled to the 1mm planned spacing:

| Contrast | Native resolution | Resampling effect |
|---|---|---|
| T2w, GRE echo-1 mag | ~1mm | Minimal change |
| DWI b=0 | ~2mm | Moderate upsampling, smooth |
| bold temporal mean | ~3mm | Heavy upsampling, blurry at 1mm |
| EPI | ~3mm | Heavy upsampling, blurry at 1mm |

Tissue contrast (which tissues are bright/dark) is inherently different across modalities — this is intentional and is what the benchmark measures. The blurriness of upsampled bold/EPI represents a genuine challenge; `SimulateLowResolutionTransform` in training partially addresses this.

### 13.3 Metric Computation in Native Space

Predictions from nnUNet are in the 1mm planned space. Metrics are computed in each contrast's **native space**:

```bash
# Resample prediction to native space (nearest-neighbour to preserve label IDs)
antsApplyTransforms -d 3 -i {prediction_1mm.nii.gz} -r {contrast_native.nii.gz} \
    -o {prediction_native.nii.gz} -n NearestNeighbor

# Compute metrics against native-space GT
nnUNetv2_evaluate_folder {gt_native_dir} {prediction_native_dir} \
    -djfile {dataset.json} -pfile {plans.json}
```

### 13.4 Metrics

Using nnUNet's built-in evaluator (`nnUNetv2_evaluate_folder`):

- **Dice:** cross-volume average (macro-average) — Dice computed per image per class, then averaged across all test images. Equal weight per subject. Excludes background class (reports classes 1–6).
- **HD95:** 95th-percentile symmetric Hausdorff distance, same aggregation (per image per class, then averaged).

Reported at three levels:
1. Per class (6 tissue classes) × per contrast × per method
2. Mean across classes × per contrast × per method (overall generalization per contrast)
3. Mean across classes and contrasts × per method (single-number comparison)

## 16. Common Pitfalls and Invariants

**Split file:** Never delete or regenerate `splits_final.json` after first use. The trainer's `do_split()` will refuse to run if the file is missing. Regeneration would produce a different split (different random seed or fold count), breaking reproducibility.

**nnUNet fold fallback:** If someone runs `nnUNetv2_train ... -f 4` with our 4-fold file, nnUNet would silently fall back to a random 80:20 split. Our `do_split()` override hard-fails before this happens.

**Brain mask for synthesis = GT tissue classes > 0:** The synthesis denormalization step uses `seg > 0` (tissue classes 1–6) as the brain mask, not `image != 0`. The T1w images in ON-Harmony are NOT skull-stripped; using the image intensity alone would include skull in synthesis. The GT segmentation (0 = skull/air/background) is the reliable brain boundary. Do not change this to an intensity threshold.

**Synthesis output range:** The synthesis output must re-enter z-score space (brain ≈ N(0,1), background = 0) before nnUNet's spatial transforms run. nnUNet's `MaskImageTransform` re-zeroes the background as a final step, but does not normalize intensity — the synthesis step must normalize first.

**Registration GT:** Non-T1w GT labels are produced by registration and are therefore subject to registration error. Quality is verified by NMI (see §6.1). Any flagged registration must be manually reviewed before including that contrast in evaluation.

**Test set immutability:** The 4 test subjects and 2 test scanner families are fixed by `00_create_splits.py`. Never retrain on test subjects or test scanners. Never use test evaluation results to select hyperparameters (val Dice on fold k is the correct model selection criterion).

**Intensity augmentation state per method:** Baseline uses all nnUNet augmentations. V26_6, SynthSeg-A, SynthSeg-B disable intensity augmentations but keep spatial augmentations + SimulateLowResolution. Mixing these up silently trains models with wrong augmentation policy.

## 18. Performance Design

### 18.1 Bottleneck: serialized pipeline with num_workers=0

GPU synthesis requires CUDA in the main thread, so `num_workers=0` is mandatory — worker processes don't share a CUDA context. Without mitigation every iteration is sequential: disk read → H2D → synthesis → crop → spatial aug → forward → backward.

### 18.2 RAM preload

At `initialize()`, before the first epoch, every training `.npz` case for the current fold is loaded into a `dict[str, dict]` in CPU RAM. ~72 volumes × ~30 MB each ≈ 2.1 GB per slot (well within system RAM across 4 slots). All subsequent batch accesses are RAM reads: disk latency = 0 for the entire training run.

### 18.3 Double-buffer prefetch thread

A single `threading.Thread` with a `threading.Queue(maxsize=2)` runs continuously alongside the training loop. It samples the next case from the RAM dict, converts to `pin_memory` tensors, and enqueues them. The main thread always finds a ready tensor; CPU prep time from RAM (~5–10 ms) is hidden behind GPU synthesis + forward/backward (~400–700 ms combined).

### 18.4 torch.compile

- `self.network = torch.compile(self.network)` — U-Net forward: ~15–25% speedup after 2–3 warmup steps.
- V26_6 synthesis function decorated with `@torch.compile(mode="reduce-overhead")` compiles K-means and Voronoi tensor ops.

All four changes live entirely in the custom trainer/dataloader — no nnUNet internals are touched.

## 17. How to Extend

### Adding a new synthesis method:
1. Add synthesizer class in `src/nnunet/transforms/`
2. Subclass `nnUNetTrainerOnHarmonyBase` in `src/nnunet/trainers/`
3. Add training script `scripts/nnunet_onharmony/03_train_{method}.sh` (same structure as existing ones)
4. Reuse Dataset030 and the existing preprocessed data — no new dataset creation

### Adding a new dataset:
1. Create splits file with the same double-isolation principle (scanner + subject exclusion)
2. Create a new nnUNet dataset (DatasetXXX) — separate from Dataset030
3. Subclass `nnUNetTrainerOnHarmonyBase` with dataset-specific label names and count
4. The split file anti-contamination checks are inherited; verify that test subjects are not in imagesTr

### Comparing across datasets:
The shared metrics format (`eval_long.csv` with columns `method, dataset, contrast, class, dice, hd95`) allows direct aggregation across datasets. `05_aggregate_results.py` accepts a `--dataset` flag for this purpose.
